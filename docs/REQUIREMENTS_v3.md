# namioshi v3 要件

この文書はv3の要求事項の正本である。現在実装はv2相当であり、v3は未実装である。

## ゲームの目的

namioshiは、最大3回のタップで波の発生位置を選び、反射経路を予測して高得点を狙う10秒ゲームとする。単にタップすれば自動的に大量得点できるゲームにはしない。

プレイヤーの判断対象は次とする。

- どこをタップするか
- どの壁またはガラス片を使うか
- 3回のタップをどの順番で使うか
- 動くビーコンの未来位置をどう読むか

## 軽量化しても変えない部分

- 最大3タップ
- 10秒
- 波の広がり
- 壁反射
- ガラス反射
- 3つのビーコン
- 反射を使うほど高得点
- タップだけで遊べること
- 公式モードでは全員が同じ条件であること
- 入力直後に波が見えること
- 通信失敗でも結果を確認できること

## 性能目標

「60fps固定」とはしない。ブラウザの表示更新に合わせ、経過時間で動きを計算する。

目標:

- 60Hz端末では滑らかな60fpsを目標にする。
- 継続して30fpsを下回る場合は公開不合格とする。
- タップから波が見えるまで50ms以内を目標にする。
- プレイ中の50msを超える処理は調査対象とする。
- 意図しない100ms以上の停止は公開停止条件とする。
- 高更新端末でもゲーム時間や波速度を変えない。
- 画面更新回数ではなく経過時間で物理を進める。
- HIGH / MID / LOWの品質段階を持つ。
- 低下した品質を同じプレイ中に頻繁に上下させない。

固定の総容量上限は設けない。代わりに次を記録・検査する。

- 初回表示までの時間
- ゲーム開始に必要な素材の読み込み時間
- 最も重い反射場面
- 最大表示波数
- 最大粒子数
- 連続再挑戦10回
- 30分連続稼働
- バックグラウンド移動と復帰10回
- WebGLコンテキスト消失と復帰
- オフラインと再接続
- 周回ごとに配列、イベント、タイマー、描画資源が増え続けないこと
- 通常想定より負荷を20％増やした試験
- 同じ公開候補版で3回連続して合格すること

## 容量と素材

3MB以下を受け入れ条件にしない。容量は報告値として記録するが、総容量だけを理由に不合格にしない。

次は引き続き検査対象とする。

- 未使用素材
- 重複素材
- 公開版ソースマップ
- 無圧縮の巨大音声
- 必要以上に大きい画像
- 外部権利を確認できない素材
- 古い公開物の残存
- 読み込まれないファイル

画像、音声、テクスチャ、Three.jsを採用する場合は、見た目または操作感へ明確な効果があることを条件にする。複数ファイル構成とES Modulesは正式に許可する。CSS、JavaScript、WebGL、Three.jsなど特定の方法だけを強制しない。
