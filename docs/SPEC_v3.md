# namioshi v3 仕様

この文書はv3ゲーム仕様の正本である。現在実装はv2相当であり、v3は未実装である。

## 状態遷移

目標状態は `BOOT`、`HOME`、`RULES`、`COUNTDOWN`、`PLAYING`、`RESULT`、`ERROR` とする。内部状態として必要なら `SUSPENDED` を追加してよい。`LOADING` は、実際に待つ必須素材がある場合だけ使う。固定時間の見せかけの `LOADING` は使わない。

基本遷移は `BOOT` → `HOME` → `COUNTDOWN` → `PLAYING` → `RESULT` とする。`RULES` は `HOME` から開き、閉じると `HOME` へ戻る。`ERROR` は、WebGLとCanvas 2Dの両方を初期化できない場合など、ゲームを継続できない時だけ使う。

カウントダウンは `3: 600ms`、`2: 600ms`、`1: 600ms`、`START: 400ms` とする。カウントダウン中の二重開始を防ぐ。HTMLのidは全画面を通して一意にする。

## HOME

最低限、正式タイトルnamioshi、1～2行の短い説明、ランキング名入力、ゲーム開始、ルール説明、ゲームをシェア、実験場へ戻る、音のON/OFF、ランキング名が公開されることの説明を置く。開始時は入力欄をblurし、iPhoneのキーボードを閉じてからカウントダウンへ進む。

## ゲーム内部座標

ゲーム内部は360×640の固定論理座標を使う。端末表示では、縦横比を保って拡大縮小する。余った領域は水面背景で埋める。ゲーム領域を切り取らない。

入力座標は次の順で変換する。

1. canvasの画面上の位置を取得する。
2. 表示倍率と余白を除外する。
3. 360×640の論理座標へ変換する。
4. ゲーム領域外の入力を拒否する。

画面サイズが変わっても、同じ公式配置と同じ論理座標タップなら同じ結果になることを要求する。

## 入力

- Pointer Eventsを使う。
- `pointerdown` へ統一する。
- `touchstart` を重複登録しない。
- `isPrimary` が真の入力だけ受ける。
- `PLAYING` 中だけ受ける。
- UI上のタップをゲーム入力にしない。
- 最大3回とし、4回目以降は無視する。
- 1入力から根の波を1つだけ作る。
- `pointercancel` を処理する。
- マルチタッチをゲーム操作に使わない。
- ピンチズーム、ダブルタップ拡大、横スクロールを抑える。
- リンクと入力欄は必要な標準操作を残す。

## モード

公式モードはランキング対象であり、全員が同じビーコン、ガラス、初期速度、配置条件で競う。同じclient versionでは同じ条件とし、ランダム配置を使わず、結果確定後に1回だけ送信する。

練習モードはランキング対象外であり、ランダム配置を許可する。Supabaseへスコアを送らず、結果画面に「練習結果」と表示する。初回v3公開でモード切り替えを実装するかは、`IMPLEMENTATION_PLAN_v3.md` のPhaseで分ける。

## 波の識別

各プレイヤータップに `rootTapId` を付ける。各波は最低限、`waveId`、`rootTapId`、`parentWaveId`、`originX`、`originY`、`radius`、`width`、`speed`、`age`、`lifetime`、`energy`、`reflectionDepth`、`reflectedBy`、`surfaceHistory`、`hitCandidates` を持つ。

反射回数は最大2回とする。同じ波が、反射元の壁またはガラス片で次のフレームに即座に再反射しないようにする。反射元を `surfaceHistory` へ記録し、十分な移動距離を消費するまで同じ面で再反射させない。

## 反射とエネルギー

初期エネルギーは `1.0` とする。壁反射後は直前のエネルギー×`0.72`、ガラス反射後は直前のエネルギー×`0.84` とする。2回反射後も同じ規則で減衰する。

エネルギーは、波の明るさ、波の帯の太さ、残像、ビーコンへ与える見た目上の揺れへ反映する。エネルギーが低いことだけを理由に、得点判定位置をずらさない。

## 得点契約

v3の得点候補値の初期契約は、直接波20点、壁で1回反射100点、ガラスで1回反射180点、2回反射300点とする。

命中精度:

```text
accuracy = clamp(1 - error / hitBand, 0, 1)
```

候補得点:

```text
candidateScore = round(baseScore * (0.8 + accuracy * 0.4))
```

同じ `rootTapId` と同じ `beaconId` の組み合わせでは、最も高い `candidateScore` だけを採用する。すでに低い候補得点を加算済みで、後から高い候補が出た場合は、差額だけを加算する。

例:

- 以前の最高候補: 20
- 新しい候補: 174
- 追加点: 154

公式スコアでは、全体コンボ倍率を使わない。公式スコアの理論上限候補は `3タップ × 3ビーコン × 2回反射の最大候補360点 = 3240点` とする。この3240点は、公式配置と得点式の試験後に確定する。不可能な値であることが判明した場合は、理由を文書へ残し、実装前に改訂する。現行ランキングに旧スコアが存在する場合、v3スコアと無条件に混ぜない。

## ビーコン

ビーコンは3個とする。公式モードでは初期位置と動きが固定で、端末サイズに依存しない。波が通過した時だけ短く揺れ、速度へ際限なく力を足さず、ばねと減衰で基準軌道へ戻る。見た目と当たり判定中心を一致させ、色だけで状態を伝えない。

## 公式配置

公式配置の正確な座標は、今回推測して作らない。決定工程は次とする。

1. 3つ以上の公式配置候補を作る。
2. 同じ3タップ候補で自動検査する。
3. 直接波だけで高得点になりすぎないことを確認する。
4. 壁反射とガラス反射を使う意味があることを確認する。
5. iPhone SE級でビーコンとガラスが重なりすぎないことを確認する。
6. 人が実際にプレイして1つを選ぶ。
7. 公式配置IDとルール版を固定する。

## 結果画面

最低限、最終スコア、NEW BESTまたは端末ベスト、プレイ回数、直接波得点、壁反射得点、ガラス反射得点、2回反射得点、ランキング送信状態、もう一度、結果をシェア、詳細ランキング、ゲーム終了、実験場へ戻るを置く。通信失敗中でも、再挑戦、シェア、終了、実験場への移動を使える状態にする。

## 端末内保存

保存候補は `namioshi.displayName`、`namioshi.bestScore`、`namioshi.playCount`、`namioshi.settings.sound`、`namioshi.lastClientVersion` とする。実際に保存していない場合、「保存されました」と表示しない。未送信スコアを自動で後日再送信する機能は、二重送信対策が決まるまで実装しない。

## シェア

シェア文の正式形は次とし、文末は必ずゲームURLにする。

```text
namioshiで {score}点！
反射する波をビーコンに重ねた。
https://chameleonjp.codeberg.page/namioshi/
```

順序は `navigator.share`、クリップボードコピー、手動コピー用テキスト表示とする。共有画面をキャンセルしただけの場合、勝手にコピーへ切り替えない仕様を検討事項として残す。

## ランキング契約

送信先:

```text
POST {SUPABASE_URL}/rest/v1/rpc/submit_score
```

ヘッダー:

```text
Content-Type: application/json
apikey: {SUPABASE_PUBLISHABLE_KEY}
```

Publishable keyを `Authorization: Bearer` へ入れない。

本文:

```json
{
  "p_display_name": "プレイヤー名",
  "p_game_slug": "namioshi",
  "p_score": 1230,
  "p_client_version": "namioshi-web-3.0.0-official001"
}
```

禁止事項は、secret key、service_role key、`public.game_scores` への直接INSERT、独自のゲーム専用RPC、共通 `submit_score` RPCの置き換え、Publishable keyの実値をREADME・仕様書・コミット・PR本文・完了報告へ転記すること、サーバー応答全文やキーを画面またはconsoleへ出すこと。

送信条件は、公式モードの結果だけ、`RESULT` を先に表示する、1プレイにつき1回、10秒程度のタイムアウト、`playId` で古い通信結果を無視、練習では送らない、通信失敗でゲーム結果を失わないこと。状態は未設定、送信中、成功、失敗を持つ。

Supabase URLとPublishable keyは、作業環境で提供されている正式値を確認する。値を推測しない。今回の文書作業では、コードの値を変更しない。

未確認事項として、現在の正式Supabase URL、Publishable keyとURLが同じプロジェクトのものか、`submit_score` の実RPC疎通、`public.games` へのnamioshi登録、現在の旧ランキング件数、v3スコアを旧ランキングと分ける方法、実験場トップの登録方式、詳細ランキングの登録方式を分けて扱う。

## WebGLとCanvas 2D

純粋WebGLを本命描画として維持する。Canvas 2Dを正式なフォールバックとして維持する。ゲームロジックは描画方式に依存させない。描画側は、ゲーム状態を受け取り、表示するだけにする。

WebGLでは、シェーダー作成失敗、プログラムリンク失敗、バッファ作成失敗、WebGLコンテキスト消失、WebGLコンテキスト復帰、高解像度の負荷、描画資源の破棄、画面サイズ変更、画面回転、低性能時の品質低下を考慮する。ブラウザ名や端末名だけで描画方式を決めず、必要な機能を実行時に確認する。

Three.jsへ移行する場合は、別作業で見た目、入力反応、iPhone SE級のフレーム時間、連続再挑戦、WebGLコンテキスト復帰、ビルドの保守性、Canvas 2Dへの切り替えを比較する。Three.jsを使うこと自体を品質の証明にしない。

## 視覚表現

目標は、暗い水面を上から斜めに見た2.5D表現である。

- 水面: 青黒、中央を少し明るくする、画面端は暗くする、細かい常時波を2層まで使う。
- 直接波: 水色、細め、残像は少ない。
- 壁反射: 青紫、直接波より残像を増やす。
- ガラス反射: 紫、短い虹色または白い縁を持つ。
- 2回反射: 金色、最も強い発光、得点価値が高いことを色以外にも太さで示す。
- ビーコン: 外側の柔らかい光、脈動する中間リング、白い中心、命中時に短く拡大する。
- ガラス片: 半透明の紫青、中心線と外側の光を分ける、両端を細く光らせる。
- ヒット: 粒子、短い光、+得点表示、DIRECT / WALL / GLASS / DOUBLEの種別表示。

WebGLの `lineWidth` だけへ太さを依存させない。波は内側と外側を持つ帯として `TRIANGLE_STRIP` で描く方針を後続Phaseへ記載する。色だけで得点種類を伝えない。

## 音

音声ファイルまたはWeb Audio APIを利用してよい。容量だけを理由に、音を禁止しない。最低限の音候補は、タップ、壁反射、ガラス反射、ビーコン命中、2回反射命中、結果表示とする。初回のユーザー操作後に音声機能を有効化する。音が利用できない場合もゲームを止めない。音量設定を端末内へ保存できる仕様にする。
