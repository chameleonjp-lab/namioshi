## 1. この調査の目的

公式モードで全員が同じ条件を使えるようにする前に、複数の配置候補を同じ方法で比較する。

このPhaseでは候補を自動採用しない。数値で確認できる部分と、実機で人が確認する部分を分ける。

## 2. 候補

| 候補 | ID | ねらい |
|---|---|---|
| 候補A・交差流 | `candidate-a-cross-current` | 中央の斜めガラスを軸に、壁反射とガラス反射を広く探す。 |
| 候補B・段流路 | `candidate-b-stair-channel` | 上から下へ段階的に反射点を置き、順序のある読み筋を作る。 |
| 候補C・開港型 | `candidate-c-open-harbor` | 中央を広く空け、壁反射と外周寄りのガラス反射を見分けやすくする。 |

すべての候補は、360×640の論理座標、ビーコン3個、ガラス片4個、固定された初期速度を持つ。

候補データには`selected`や`official`を持たせていない。採用前の比較候補であることをコードでも強制する。

## 3. 比較条件

### 調査地点

```text
X: 30から330まで30刻み
Y: 90から590まで50刻み
合計: 121地点
```

### 到達時刻

```text
0.25秒から3.00秒
0.05秒刻み
合計: 56時刻
```

### 共通3タップ

候補間の比較用として、次の3地点をすべての候補へ同じ順序で使う。

```text
1: (90, 140)
2: (180, 340)
3: (270, 490)
```

### 反射面

- 画面の4辺
- 候補ごとのガラス片4個
- 直接波
- 1回の壁反射
- 1回のガラス反射
- 異なる面を使う2回反射

### 比較用の波条件

```text
波速度: 165論理px/秒
波寿命: 3秒
命中許容幅: 18論理px
```

得点の参考値にはv3仕様書の候補得点を使うが、本番の得点式はこのPhaseで変更していない。

## 4. 分析方法

反射経路は、反射面の向こう側へ目標を写して直線経路へ置き換える方法で調べる。ガラス片は有限の線分として扱い、延長線だけへ当たる経路は数えない。

ビーコンは初期位置と速度から10秒間の位置を計算し、画面端で反転する。候補ごとに同じ時刻刻みと同じタップ地点を使う。

「経路数」は、タップ地点、ビーコン、反射面の組み合わせについて、56時刻のどこかで許容幅へ入った組み合わせの数である。同じ経路を毎フレーム分だけ重複して数えない。

## 6. 数値から確認できること

### 候補A

ガラス反射と2回反射の候補が3案中で最も多い。一方、ビーコンとガラスがほぼ重なる時刻があり、近接警告が2組ある。ビーコン同士の最小間隔も3案中で最も小さい。

### 候補B

壁反射の候補は十分にあるが、ガラス反射の候補は3案中で最も少ない。共通3タップの参考得点も最も低い。反射経路が1地点へ集中する割合は3案中で最も低い。

### 候補C

ビーコン同士の最小間隔が3案中で最も広い。ガラス反射の候補は候補Aに近い。ガラス端点が画面端へ最も近いため、iPhone SE級での見やすさは実機確認が必要である。

これらは観察結果であり、採用の推薦ではない。

## 7. この分析だけでは決められないこと

- 波を押した直後に経路を理解できるか。
- 反射が偶然ではなく自分の判断に見えるか。
- 3回のタップで考える時間が足りるか。
- ガラス片とビーコンをiPhone SE級で見分けられるか。
- 同じ攻略だけが最適にならないか。
- 実際のWebGL表現と音を加えた時に混雑しないか。

数値上の経路が多くても、画面が読みづらければ公式配置には使わない。

## 9. 人が確認する項目

採用判断では、最低限次を確認する。

1. iPhone SE級で3つのビーコンと4つのガラス片を区別できるか。
2. 直接波だけでなく、壁反射とガラス反射を試したくなるか。
3. ビーコンとガラスが重なる場面を理不尽に感じないか。
4. 3タップの選択肢が一つに固定されすぎないか。
5. 候補A、B、Cの採用、または修正版候補Dの作成を明示できるか。

## 10. 採用手順

この文書の状態は`human-decision-pending`である。

ユーザーが候補を明示するまで、次を行わない。

- 候補を`src/game/layouts.js`へ移す。
- 公式配置IDを`config.js`へ設定する。
- 公式モードで候補を使う。
- ランキング対象にする。
- 他候補を削除する。

候補が選ばれた後、別Pull Requestで選択記録だけを確定し、その後Phase 3Cで公式モードと練習モードを分ける。

## 11. 分析の再実行

```bash
npm run analyze:layouts
```

候補を意図的に変更した場合だけ、次で保存結果を更新する。

```bash
npm run analyze:layouts:write
```

保存結果が変わった時は、数値差と理由をレビューし、機械的に受け入れない。
