# namioshi v3 実装計画

v3は一度にすべて実装しない。各Phaseを別Pull Requestにし、1つのPull Requestで得点、ランキング、WebGL、ビルド、画面を同時に変更しない。現在実装はv2相当であり、v3は未実装である。

## Phase 1: 現行コードの正しさ

対象候補:

- 重複しているHTML idの確認と修正
- ルールボタンの動作
- 見せかけのLOADINGの削除
- カウントダウン二重起動防止
- 「保存されました」表示と実保存の整合
- 共有キャンセルの扱い
- 現在のSupabase設定差異の確認
- 現行コードの構文と起動試験

このPhaseでは、得点式と見た目を変更しない。

## Phase 2: 開発ソースとビルドの整理

対象候補:

- 実質JavaScriptの `.ts` を `.js` へ整理
- 不要なvendored TypeScript shimの削除
- 使っていないVite/Three.js代替依存の確認
- `src` を正本、`dist` を生成物として一方向化
- 手作業による二重管理の解消
- ビルドの再現性
- 静的配信確認

このPhaseでは、ゲームルールと描画結果を変えない。

## Phase 3: 固定論理座標と公式配置

対象候補:

- 360×640論理座標
- 入力座標変換
- アスペクト比維持
- 公式配置候補
- 練習ランダム
- 端末サイズ差の解消
- 同じ入力で同じ結果になる試験

このPhaseでは、得点式をまだ変更しない。公式配置は、3つ以上の候補を作り、同じ3タップ候補で自動検査し、直接波だけで高得点になりすぎないこと、壁反射とガラス反射を使う意味があること、iPhone SE級でビーコンとガラスが重なりすぎないことを確認し、人が実際にプレイして1つを選び、公式配置IDとルール版を固定する。

## Phase 4: 反射と得点

対象候補:

- `rootTapId`
- `parentWaveId`
- `surfaceHistory`
- 即時再反射防止
- エネルギー減衰
- 同一タップ・同一ビーコンの最高候補方式
- 全体コンボ倍率の廃止
- 得点内訳
- 理論上限試験

このPhaseでは、Supabaseへv3スコアを送信しない。

## Phase 5: ランキング契約

対象候補:

- 正式Supabase URL確認
- Publishable key確認
- `apikey` のみ
- 共通 `submit_score`
- 4項目本文
- `playId`
- 送信タイムアウト
- 公式だけ送信
- 初回とベスト
- 旧スコアとの分離
- `public.games`
- 実験場トップ
- 詳細ランキング

本番ランキングを汚す架空スコアを勝手に送らない。SQL変更が必要な場合は、別作業として案を提示し、ユーザー承認前に実行しない。

## Phase 6: 画面とアクセシビリティ

対象候補:

- HOME
- RULES
- COUNTDOWN
- PLAYING
- RESULT
- 名前入力
- キーボードを閉じる処理
- 結果内訳
- 実験場リンク
- 詳細ランキングリンク
- safe-area
- 320×568
- 横画面
- aria-live
- focus-visible
- reduced motion
- 音設定

このPhaseでは、物理と得点を変更しない。

## Phase 7: 高品質WebGLと音

対象候補:

- 水面の陰影
- 波を帯で描く
- 反射種類ごとの見分け
- ガラスの光
- ビーコン3層
- ヒット文字
- 粒子
- 効果音
- 品質段階
- 描画資源の再利用
- WebGLコンテキスト消失と復帰
- Canvas 2Dの視覚整合

このPhaseでは、ランキング式を変更しない。

## Phase 8: 性能、実機、公開

対象候補:

- 単体テスト
- ブラウザ自動テスト
- 性能測定
- 連続再挑戦
- バックグラウンド復帰
- オフライン
- 実Supabase
- Codeberg Pages
- 実験場トップ
- 詳細ランキング
- シェアURL
- ロールバック手順
- 公開候補版3回連続合格
