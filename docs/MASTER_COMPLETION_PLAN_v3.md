## 1. 完成の定義

v3は、コードが動くだけでは完成としない。次の条件をすべて満たした時だけ完成とする。

### 1.1 ゲームとして完成している

- 1プレイは10秒で終了する。
- プレイヤーが発生できる根の波は最大3回である。
- 4回目以降のゲーム入力は無視される。
- 壁反射とガラス反射を使う意味がある。
- 直接波だけを無造作に置いて高得点を取れる設計ではない。
- 反射回数は最大2回である。
- 同じタップ由来の波と同じビーコンの組み合わせで、得点が重複しない。
- 公式モードでは全員が同じ配置、同じ初期条件、同じ得点式で競う。
- 練習モードのランダム結果はランキングへ送られない。
- 同じ公式配置と同じ論理入力では、端末の画面サイズが違っても同じ結果になる。

### 1.2 操作と画面が完成している

- `HOME / RULES / COUNTDOWN / PLAYING / RESULT / ERROR` が破綻なく遷移する。
- 名前未入力では公式モードを開始しない。
- 入力後にiPhoneのキーボードがゲーム画面へ残らない。
- 320×568 CSSピクセルで、必要なボタンが見切れない。
- safe-area、縦画面、横画面、画面回転へ対応する。
- 結果画面で、最終得点、端末ベスト、プレイ回数、得点内訳、送信状態を確認できる。
- 通信が失敗しても、再挑戦、シェア、終了、実験場への移動を使える。
- 色だけに依存せず、波の種類と得点価値を区別できる。

### 1.3 描画と音が完成している

- WebGL版で、水面、直接波、壁反射、ガラス反射、2回反射、ビーコン、ガラス片、命中演出を見分けられる。
- WebGLの線幅だけへ波の太さを依存させない。
- Canvas 2D版でも、同じゲームルールと必要な視認性を維持する。
- WebGL初期化失敗時にCanvas 2Dへ切り替わる。
- WebGLコンテキスト消失後に復帰するか、安全にCanvas 2Dへ切り替わる。
- タップ、反射、命中、結果の音が区別できる。
- 音を利用できない環境でもゲームが止まらない。
- 音のON/OFFを保存できる。

### 1.4 ランキングが完成している

- 共通 `submit_score` RPCだけを使う。
- `game_slug` は `namioshi` である。
- 公式モードの結果だけを送る。
- 1プレイにつき送信は1回だけである。
- Publishable keyは `apikey` ヘッダーへだけ設定する。
- Publishable keyを `Authorization: Bearer` へ入れない。
- secret keyとservice_role keyを公開物へ入れない。
- 送信は10秒程度で打ち切る。
- 古いプレイの通信結果が、新しい結果画面を書き換えない。
- v2以前の得点とv3得点を、無条件に同じ順位表へ混ぜない。
- Supabase URL、Publishable key、RPC、`public.games` 登録が同じプロジェクトであることを実通信で確認する。

### 1.5 性能と安定性が完成している

- 60Hz端末では滑らかな60fpsを目標にする。
- 継続して30fpsを下回る状態を公開しない。
- タップから最初の波表示まで50ms以内を目標にする。
- 意図しない100ms以上の停止を公開停止条件とする。
- 高更新端末でも、ゲーム時間、波速度、得点判定が変わらない。
- 連続再挑戦10回で、イベント、タイマー、配列、描画資源が増え続けない。
- 30分連続稼働で、操作不能、表示崩れ、音の多重化、通信の多重化が起きない。
- バックグラウンド移動と復帰を10回繰り返しても復帰できる。
- オフライン時に結果画面が操作不能にならない。
- 同じ公開候補版で、主要検収を3回連続して通す。

### 1.6 公開作業が完成している

- 公開元となる正しいコミットとタグが特定されている。
- `src` が正本で、`dist` は同じコミットから再生成できる。
- Codeberg Pagesで実際に起動する。
- 実験場トップと詳細ランキングから到達できる。
- シェア文の末尾URLからゲームへ到達できる。
- 公開前のバージョンへ戻す手順が文書化されている。
- 使用素材の出所と利用条件を確認している。
- 実機未確認項目が、必須項目として残っていない。

## 3. 作業管理の基本規則

### 3.1 1つのPull Requestで扱う目的

1つのPull Requestは、1つの目的だけを扱う。次の変更を同時に行わない。

- ビルド整理とゲームルール変更
- 固定座標と得点式変更
- 得点式変更とランキング本番送信
- ランキング変更と画面全面改修
- WebGL全面改修と物理変更
- 実機検収と未関連の新機能追加

### 3.2 Git運用

- `main` へ直接pushしない。
- 最新の統合状態から作業ブランチを作る。
- Pull RequestはDraftで作る。
- タイトルへ「Draft」と書くだけでなく、GitHub上のDraft状態を確認する。
- 自動マージを設定しない。
- ユーザー確認前にマージしない。
- force pushを行わない。
- 由来不明の未コミット変更を削除、上書き、stash、resetしない。
- 生成物を更新するPull Requestでは、正本ソースと生成物の差を説明する。

### 3.3 各Pull Requestの開始条件

- 基準ブランチと基準コミットが特定されている。
- 作業開始時の未コミット差分が記録されている。
- 変更対象ファイルと変更禁止ファイルが先に列挙されている。
- 前段階の必須ゲートが通っている。
- 不明な値を推測しなくても作業できる。
- 本番データを変更する場合は、ユーザーの明示承認がある。

### 3.4 各Pull Requestの完了条件

- 目的外の差分がない。
- 変更した機能の自動検査がある。
- 既存の `build / verify / test` が通る。
- 実施できなかった確認を「成功」と書いていない。
- 実機確認が必要な項目は未確認のまま残している。
- 変更前へ戻す方法が説明されている。
- Pull Request本文に、変更、非変更、検証、未確認、次段階が書かれている。

### 3.5 作業停止条件

次の場合は、実装を進めず停止して報告する。

- 正本同士に解消できない矛盾がある。
- 正式なSupabase情報を確認できない。
- v3得点を旧ランキングと安全に分けられない。
- 公式配置を人が選んでいないのに本番配置を確定する必要がある。
- 変更禁止ファイルを触らないと目的を達成できない。
- 基準外の大きな未コミット変更がある。
- 必須検査が今回の変更と無関係な理由で失敗し、原因を切り分けられない。
- iPhoneで継続30fpsを下回るが、品質低下でも改善しない。
- 使用素材の権利を確認できない。

# 6. 詳細実装計画

## Phase 2A: JavaScript正本へ統一する

### 目的

実質JavaScriptである `.ts` ファイルと疑似TypeScript変換をなくし、`src` をブラウザで読める普通のJavaScript正本にする。

### 主な作業

- 実行対象の `.ts` を `.js` へ変更する。
- 相対importへ `.js` 拡張子を付ける。
- CSSをJavaScriptからimportせず、HTMLの `link` で読む。
- `src/types` の不要な実行時placeholderを削除する。
- 未使用の旧シェーダーファイルを参照確認後に削除する。
- root `index.html` を `./src/main.js` と `./src/ui/styles.css` へ向ける。
- `scripts/build.mjs` を再帰コピー式へ変える。
- build前に `dist` を削除する。
- `src` と `dist/assets` の対応ファイルをバイト単位で一致させる。
- `.ts / .tsx` が残った場合はbuildを失敗させる。

### 変更してはいけないもの

- 物理定数
- 得点式
- 配置
- UI状態
- WebGLの描画結果
- Canvas 2Dの描画結果
- Supabase設定
- 共有文

### 自動検査

- すべての `.js` へ構文検査を行う。
- すべての相対importの参照先を確認する。
- `.ts`、`import type`、型注釈、bare importを検出する。
- `src` と `dist/assets` の差を検出する。
- rootとdistのHTML参照を確認する。

### 完了条件

- `src` に `.ts / .tsx` がない。
- 疑似トランスパイルを使わない。
- 既存ゲームの条件式、数値、文字列、呼び出し順が変わっていない。
- G1の画面と共有処理が維持されている。

### 推奨ブランチ

`codex/namioshi-v3-phase2a-js-source`

## Phase 3A: 360×640固定論理座標を導入する

### 目的

端末の画面サイズが得点、波速度、距離、反射時刻へ影響しないようにする。

### 実装契約

```text
論理幅: 360
論理高: 640
表示倍率: min(viewWidth / 360, viewHeight / 640)
表示幅: 360 × 表示倍率
表示高: 640 × 表示倍率
左右余白: (viewWidth - 表示幅) / 2
上下余白: (viewHeight - 表示高) / 2
```

入力変換:

```text
logicalX = (clientX - canvasRect.left - offsetX) / scale
logicalY = (clientY - canvasRect.top  - offsetY) / scale
```

論理領域外の入力は拒否する。

### 主な作業

- `viewport.js` を追加する。
- Worldの幅と高さを360×640へ固定する。
- 画面サイズ変更でWorldを作り直さない。
- WebGLとCanvas 2Dの両方へ同じ表示変換を適用する。
- HUDとゲーム入力領域を分離する。
- 画面回転時にプレイ状態を壊さない。
- DPRは描画解像度だけに使い、物理座標へ使わない。

### 自動検査

- 320×568、375×812、390×844、1024×1366で同じ論理入力を与え、同じ波位置と得点になる。
- 余白上の入力が拒否される。
- 左上、中央、右下の変換誤差が0.25論理ピクセル以内である。
- 画面回転前後でWorldの状態が変わらない。

### 完了条件

- 端末サイズを変えても公式シミュレーション結果が一致する。
- ゲーム領域を切り取らない。
- 320×568で操作領域が視認できる。

### 推奨ブランチ

`codex/namioshi-v3-phase3a-logical-viewport`

## Phase 3B: 公式配置比較ラボを作る

### 目的

Codexが感覚だけで公式配置を決めず、複数候補を自動比較し、人が最終選択できるようにする。

### 本番へ入れない比較機能

```text
tools/layout-lab.html
tools/layout-candidates.js
tools/analyze-layouts.js
docs/OFFICIAL_LAYOUT_STUDY_v3.md
```

### 候補条件

少なくとも3候補を作る。各候補は次を固定する。

- ビーコン3個の初期位置
- ビーコン3個の基準軌道と速度
- ガラス片4個前後の位置、角度、長さ
- 公式配置ID
- ルール版

### 比較項目

- 直接波だけで得られる最大候補得点
- 壁反射が有効な場所の数
- ガラス反射が有効な場所の数
- 2回反射が理論上可能か
- ビーコンとガラスの視覚重なり
- 3タップの選択肢が一つに固定されすぎないか

## Phase 4A: 波の親子関係と反射処理を作り直す

### 目的

反射元で即座に再反射する問題、親子波の追跡不能、同じ物理現象の重複処理を解消する。

### 波データ

各波は最低限、次を持つ。

```js
{
  waveId,
  rootTapId,
  parentWaveId,
  originX,
  originY,
  previousRadius,
  radius,
  width,
  speed,
  age,
  lifetime,
  energy,
  reflectionDepth,
  reflectedBy,
  surfaceHistory,
  hitCandidates
}
```

### 反射規則

- 反射深度は0、1、2のいずれかである。
- 深度2の波は新しい反射波を作らない。
- 壁反射では仮想発生源を壁の反対側へ鏡写しする。
- ガラス反射では、線分へ射影した交点を基準に発生源を鏡写しする。
- 反射子波は、親が衝突した時点の総移動距離を引き継ぐ。
- 子波を半径1から再開しない。
- `previousRadius` と `radius` の間で面を横切った時だけ反射を発生させる。
- 親と子へ反射元の面IDを記録する。
- 同じ面は、十分な移動距離を消費するまで再判定しない。
- 同じ波と同じ面の組み合わせは1回だけ反射する。

### エネルギー

```text
根の波: 1.0
壁反射: 親energy × 0.72
ガラス反射: 親energy × 0.84
```

エネルギーは見た目へ使い、当たり判定中心をずらさない。

### 自動検査

- 左、右、上、下の壁で1回だけ反射する。
- 反射直後に同じ壁で2回目が発生しない。
- ガラス片の端点付近で重複反射しない。
- 深度が2を超えない。
- 親子で `rootTapId` が一致する。
- 子の `parentWaveId` が正しい。
- 1/60秒と1/120秒相当の描画更新で同じ固定更新結果になる。

### 完了条件

- 反射経路をログで追跡できる。
- 同じ入力で同じ反射系列になる。
- 現在の見た目を一時的に簡略化しても、判定が正しい。

### 推奨ブランチ

`codex/namioshi-v3-phase4a-wave-reflection`

## Phase 4B: 最高候補得点方式を実装する

### 目的

同じタップ由来の複数波が同じビーコンへ何度も加点する問題をなくし、反射を狙う意味を明確にする。

### 基礎点

| 種類 | 基礎点 |
|---|---:|
| 直接波 | 20 |
| 壁1回反射 | 100 |
| ガラス1回反射 | 180 |
| 2回反射 | 300 |

### 命中精度

```text
accuracy = clamp(1 - error / hitBand, 0, 1)
candidateScore = round(baseScore × (0.8 + accuracy × 0.4))
```

1候補の最大は360点である。

### 加点台帳

`rootTapId × beaconId` の3×3台帳を持つ。

```js
bestCandidates[rootTapId][beaconId] = {
  score,
  type,
  error,
  waveId,
  reflectionDepth
}
```

新しい候補が高い時だけ、差額をプレイ中スコアへ加える。

結果画面の内訳は、最終的に採用された9枠の候補から再集計する。低い候補から高い候補へ更新された場合、古い種類を最終内訳へ残さない。

同点時の表示分類は、実装前に次の順で固定する。

1. 候補得点が高い
2. 反射深度が高い
3. 命中誤差が小さい
4. `waveId` が小さい

この順序は合計得点を変えず、結果内訳を決定的にするために使う。

### 全体コンボ

公式モードでは全体コンボ倍率を削除する。練習モードでも同じ得点式を基本とし、別の演出倍率を付ける場合は得点へ加えない。

### 理論上限

暫定上限は次である。

```text
3 root taps × 3 beacons × 360 = 3240
```

実装後、列挙試験と公式配置試験で確定する。理論上限を変更する場合は、実装前に正本文書を更新する。

### 完了条件

- 同じ根の波とビーコンで加点枠が1つだけである。
- 候補更新時は差額だけが増える。
- 最終内訳の合計が最終スコアと一致する。
- 公式スコアが3240を超えない。
- 旧v2ランキングへまだ送信しない。

### 推奨ブランチ

`codex/namioshi-v3-phase4b-scoring-ledger`

## Phase 7D: WebGL消失と描画切替を完成させる

### 目的

WebGLコンテキストが失われても、ゲーム全体を再読込せずに安全に復帰させる。

### 処理

`webglcontextlost`:

- `preventDefault()` を呼ぶ。
- 入力を一時停止する。
- ゲーム状態、スコア、残り時間を保持する。
- 描画ループのWebGL呼び出しを停止する。

`webglcontextrestored`:

- シェーダー、プログラム、バッファ、テクスチャを作り直す。
- 復帰に失敗したらCanvas 2Dへ切り替える。

Canvas 2D切替:

- 同じcanvasでWebGL取得後に2D contextを取れない場合を考慮する。
- 必要なら新しいcanvasへ差し替える。
- UIとWorldは維持する。

### 完了条件

- 強制コンテキスト消失試験で復帰またはCanvas切替が成功する。
- スコアと入力回数が失われない。
- イベント登録が重複しない。

### 推奨ブランチ

`codex/namioshi-v3-phase7d-context-recovery`

### G7通過条件

MID品質で情報を明確に読み取れ、iPhone実機で性能目標を満たし、Canvas 2Dでも遊べること。

# 8. 品質測定の基準

## 8.1 ゲーム結果の決定性

同じ次の入力が同じ結果になること。

```text
公式配置ID
ルール版
固定物理刻み
3つの論理タップ座標
各タップの論理時刻
```

比較対象:

- 320×568
- 390×844
- 1024×1366
- 60Hz相当
- 120Hz相当

スコア、採用候補、反射系列が一致しなければ不合格とする。

## 8.2 フレーム時間

- 60fpsを目標にする。
- 平均値だけでなく、遅いフレームを確認する。
- プレイ中の50ms超を調査対象にする。
- 意図しない100ms超を公開停止条件にする。
- 継続30fps未満を公開不合格にする。

測定は、最も波と粒子が多い場面で行う。

## 8.3 入力反応

pointerdownの受理時刻から、次の描画で波が確認できるまでを測る。

- 目標: 50ms以内
- 100ms超: 不合格

音の遅延と描画の遅延を分けて記録する。

## 8.4 継続安定性

対応APIがある環境ではJS heapを記録する。iPhone Safariで詳細値を取得できない場合は、次を代替指標とする。

- イベント登録数
- タイマー数
- 波配列上限
- 粒子配列上限
- WebGLバッファ数
- AudioNodeの残留
- 10周目のフレーム時間
- 1周目と10周目の入力反応

## 8.5 読み込み

総容量には固定上限を設けない。代わりに次を記録する。

- HTML表示まで
- HOME表示まで
- 開始可能になるまで
- 必須素材の転送量
- 追加素材の転送量
- キャッシュ有無
- 404と再試行

HOMEに不要な高品質素材は、HOME表示後に読み込んでよい。
